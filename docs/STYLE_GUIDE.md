# Руководство по стилю кода

Этот документ — железный стандарт оформления кода и использования внутренних хелперов проекта. **Все новые модули и плагины обязаны следовать этим правилам без исключений.**

Цель — поддерживать код единообразным, читаемым и предсказуемым.

---

## 1. Стиль и структура кода

- **Шелл:** Все скрипты должны использовать `#!/bin/bash`.
- **Синтаксис:**
  - Для тестов предпочитайте `[[ ... ]]` вместо `[ ... ]`.
  - Для подстановки команд используйте `$(...)` вместо обратных кавычек (`` `...` ``).
- **Переменные:**
  - Всегда объявляйте переменные внутри функций с помощью `local`.
  - Используйте синтаксис `${variable_name}` для большей ясности и во избежание проблем с `set -u`.
- **Именование функций:**
  - Используйте `snake_case` (например, `show_my_menu`).
  - Внутренние (приватные) хелперы, не предназначенные для вызова из других модулей, должны начинаться с подчеркивания (например, `_helper_function`).
- **Форматирование:** Длинные и сложные однострочные команды **должны** быть разбиты на несколько строк для читаемости.

## 2. Генерация меню и UI (Архитектура «Манифест меню»)

### 2.1. Декларация пунктов меню

Интеграция в меню происходит **декларативно**. Вместо правок в `case`...`esac`, вы просто описываете пункты меню в специальном блоке `@menu.manifest` в любом `.sh` файле.

```bash
#
# @menu.manifest
#
# @item( PARENT | KEY | TITLE | FUNCTION | ORDER | GROUP | DESCRIPTION )
# @item( main | s | 🛡️ Безопасность | show_security_menu | 50 | 50 | Управление защитой сервера )
#
```

- **`PARENT`**: ID родительского меню (`main` для главного меню).
- **`KEY`**: Клавиша вызова.
- **`TITLE`**: Название пункта.
- **`FUNCTION`**: Функция-обработчик, которая будет вызвана.
- **`ORDER`**: Порядок сортировки.
- **`GROUP`**: ID для визуальной группировки.
- **`DESCRIPTION`**: Опциональное описание.

> За подробным описанием формата обратитесь к **`docs/GUIDE_MODULES.md`**.

### 2.2. Структура функции-меню

Функция, которая служит "контейнером" для подменю, теперь предельно проста. Она не рисует пункты сама, а делегирует это генератору.

```bash
show_security_menu() {
    # "security" - это ID, который будет PARENT'ом для дочерних пунктов.
    local menu_id="security" 
    
    enable_graceful_ctrlc
    while true; do
        clear
        menu_header "🛡️ Меню безопасности"
        printf_description "Этот раздел содержит утилиты для укрепления защиты сервера."

        # 1. Генератор сам находит и отрисовывает все пункты
        #    с манифестом "@item( security | ... )"
        render_menu_items "$menu_id"

        printf_menu_option "b" "Назад"

        local choice
        choice=$(safe_read "Ваш выбор: ") || break
        
        # 2. Выход из меню
        if [[ "$choice" == "b" || "$choice" == "B" ]]; then
            break
        fi

        # 3. Получаем и выполняем действие для выбранного пункта
        local action
        action=$(get_menu_action "$menu_id" "$choice")
        
        if [[ -n "$action" ]]; then
            eval "$action"
        else
            printf_error "Нет такого пункта." && sleep 1
        fi
    done
    disable_graceful_ctrlc
}
```
- **Цикл:** Всегда `while true`, обернутый в `enable_graceful_ctrlc` / `disable_graceful_ctrlc`.
- **Отрисовка:** `render_menu_items "menu_id"` рисует все дочерние пункты.
- **Обработка:** `get_menu_action "menu_id" "$choice"` получает команду для `eval`.

## 3. Взаимодействие с пользователем

### 3.1. Вывод сообщений

Никогда не используйте `echo` для сообщений пользователю. Используйте стандартные хелперы:
- `info "Обычное сообщение"`
- `ok "Операция выполнена успешно"`
- `warn "Предупреждение о чем-либо"`
- `err "Произошла ошибка"` (автоматически добавляет паузу)
- `debug_log "Отладочное сообщение"` (выводится только при `DEBUG_MODE=on`)

### 3.2. Ввод данных

- **Простой ввод:** `my_var=$(safe_read "Введите значение:") || break`
- **Подтверждение (Да/Нет):** `if ask_yes_no "Вы уверены?"; then ... fi`
- **Непустой ввод:** `my_var=$(ask_non_empty "Введите имя:") || break`
- **Число в диапазоне:** `my_port=$(ask_number_in_range "Введите порт" 1 65535) || break`
- **Ввод пароля (скрытый):** `local sudo_pass; sudo_pass=$(ask_password "Пароль sudo:") || break`
- **Пауза:** `wait_for_enter`

### 3.3. Цвета

- Все цвета определены как переменные `${C_*}` в `modules/core/common.sh` (например, `${C_GREEN}`, `${C_GRAY}`).
- **Запрещено** использовать "сырые" ANSI-коды (`\033[...`).
- Для сброса цвета всегда используйте `${C_RESET}`.

## 4. Работа с системой

- **Выполнение команд:**
  - Всегда используйте `run_cmd` для системных команд, требующих `sudo` (`apt`, `systemctl`, `sed` для системных файлов).
  - `run_cmd systemctl restart nginx`
- **Логирование:**
  - Пишите в лог-файл проекта через `log "Мое сообщение для лога"`.
- **Конфигурация:**
  - Читайте и пишите в `config/server.conf` только через `get_config_var "KEY"` и `set_config_var "KEY" "VALUE"`.
- **Зависимости:**
  - Для проверки и установки утилит (`curl`, `jq`) используйте `ensure_dependencies "curl" "jq"`.

## 5. Генерация дочерних скриптов (Heredoc)

При генерации скриптов (например, для `systemd`) с помощью `cat << EOF`, соблюдайте следующие правила, чтобы избежать ошибок `Exec format error` и `unbound variable`.

-   **Правильное экранирование:**
    -   Чтобы переменная была вычислена **при запуске дочернего скрипта**, экранируйте `$` (`\$VAR`, `\$(command)`).
    -   Чтобы переменная была вычислена **в момент генерации**, не экранируйте ее (`$VAR`).
    -   **Никогда** не используйте двойное экранирование (`\\$`).

-   **Область видимости:** Не используйте `local` в глобальной области видимости генерируемого скрипта.

-   **Устойчивость к `set -u`:** Объявляйте переменные (например, `MY_VAR=""`) до `if`-блоков, если они используются после них.

> Подробности и примеры см. в `WARP.md`, стандарт №11.
